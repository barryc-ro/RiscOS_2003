2.1 Mouse control

2.1.1 NC OS 1.06/1.15/1.16

Mouse users are able to scroll the page in all directions by clicking the mouse button on a web page and dragging the mouse pointer in the direction they wish to scroll. The history list is implemented as an HTML page which is created and displayed when the user clicks on the history list button on the button bar or chooses the 'Show page history' menu option. 3.0 Implementation notes

Here are notes on the implementation or limitations of various standard features.

3.1 Server push

Not in NC 1.06

Server push is accomplished by an HTTP server keeping an open connection to the browser and sending it multiple files embedded in one HTTP stream.

This has two uses.

  1. Automatically reloading an HTML page
  2. Image animation

If a link is followed to a server-push object then whenever the server sends a new object the whole page (which may be HTML, text or an image) refreshes - in the same way that client-pull refreshing is performed.

If an inline image is specified as a server-push object then when a new image is received in the stream it replaces the current one. Thus giving simple animation.

Note any object could be a server-push object. Thus it would be possible to give a server-push object to, say, BGSOUND and change the sound being played from time to time. The browser does not support server-push on embedded objects in this release.

3.2 Font colour

Not in NC 1.06

This implementation has a limit of 16 different font colours on a page.

It is possible to change the font colour for every character on a page. However in practice only a few colours are used on a single page.

The value 16 is determined by the space currently available in the data structures. To remove the limits on this value requires either increasing the memory overheads or a different strategy requiring more work.

This limit does not include the colours set for the whole page (the plain text, link, visited link and activated link colours), the actual limit is 15 individual colours plus one colour meaning 'use as dictated by page defaults'

Note hyperlinks are not affected by the FONT COLOR tag unless it is set within the anchor tags. ie. <FONT COLOR=red> <A HREF="#top"> link to top </A> </FONT> would be displayed in the default link colour. Wheres <A HREF="#top"> <FONT COLOR=red> link to top </FONT> would be displayed in red. This is the apparent behaviour of Netscape 4 on the PC.

When the limit is exceeded then the closest colour from those already allocated is used.

3.3 Background sound

Not in NC 1.06

Format: <BGSOUND SRC="URL" LOOP=[integer|INFINITE]>

When a BGSOUND tag is encountered on a page then the URL referenced is downloaded and played. If specified then whilst the page is open the sound referenced is looped. When the page is left then the sound stops.

A sound can be played that will not stop when the page is left by using the existing client-pull refresh mechanism.

The actual playing is handled by the MediaPlay application. To allow the sound to be linked to the page, so that it stops when the page is left, it is actually implemented as a plugin. This also has the side-effect that <EMBED SRC=XXX.wav> and similar constructs are supported.

MediaPlay supports the playback of the common internet sound formats such as Wave, AIFF, AU and MIDI as well as movie files and any format that can be controlled through the Acorn Replay infrastructure.

3.4 Secure Sockets Layer

3.4.1 Secure transport icon

Not in NC 1.06

An icon on the toolbar shows whether the current page was loaded over a secure or unsecure link.

3.4.2 User feedback

If an SSL authentication cannot be made then the user is not aware of this. The transaction still goes ahead and is still encrypted but the user is not aware that the site contacted cannot be trusted.

3.4.3 Managing certificates

There is no user control over certificates in this release. The list of available authorities is fixed either in ROM or on the server and is only under server control. The user is not able to add or remove certificates.

3.5 Parser features

Not in NC 1.06

3.5.1 Comment backtracking

Comment parsing is changed to work in two stages. In the first stage only valid comments are recognised (<!-- -->). If, at the end of parsing the whole document, a comment is still open, then the parser backtracks to the start of the comment and looks for the first occurence of > or < and closes the comment on or before that character respectively.

3.5.2 Character and numeric entities

The entity parser is extended to be much more forgiving of badly constructed entities.

  • When matching entity names the initial search is case-insensitive.
  • If the entity found does in fact have separate upper and lower case versions then the case of the first character of the entity as written is used to determine the case of the result.
  • Numeric entities are recognised even if they are missing the # (eg &160; is recognised as a hard space).
  • Unmatchable entity names are displayed as plain text (eg &fred;)

3.5.3 Tag and attribute names

The name parser is changed to work on a closest rather than an exact match for element and attribute names and values. This more closely resembles Netscape's handling.

3.6 Frames

Frames are a method of displaying multiple HTML files on screen in a controlled fashion. Each document can be independently scrolled and the space for each document may be reallocated by the user.

The browser uses the standard WIMP furniture for scroll bars.

The cursor link highlighting algorithm is modified to move the cursor from frame to frame whilst following links. If the highlight reached the bottom of a frame it will move to the top of the next frame and similarly when moving backwards.

3.7 Tables

Tables are implemented according to the latest available W3C draft specification (currently 23-Jan-96). This draft contains methods to allow correct rendering of tables written to display on existing browsers.

3.8 Client side image maps

Client-side image maps are supported according to the internet draft document referenced. Selecting an image map attempts to use a local description, degrading to use of a server side imagemap if necessary (and available) without user knowledge. Status information on the URL referenced by the area under the pointer is displayed in the status bar.

3.9 Client side cookies

Cookies are pieces of information that link a site or portion of a site with a user. They need to be stored when received, possibly kept across sessions, and sent back to the server when the approriate URLs are accessed.

Cookies are stored in a file referenced by the environment variable NCFresco$Cookies, typically in the users home directory. This file is updated whenever a new cookie is received or one expires. The limits on how many cookies should be held at one time are defined in the spec referenced. This is checked when a new cookie is added and cookies are expired on a least recently used basis.

3.10.1 NC OS 1.06

The 16 Windows VGA palette colour names are understood as well as the standard RGB format

3.10.2 NC OS 1.15

The extended Unix colour name set is also understood (in english and american forms) as well as colours of the form greyXX.

4.1.1.1 Replay movie playback

A mechanism whereby Replay movies can be embedded and played in HTML pages is introduced. When the browser sees the HREF ncfrescointernal: it knows that what follows is an instruction to the browser rather than a real URL. 4.2 Audio playback

4.2.1 NC OS 1.06 - Soundplay

The Web browser uses a MIME mapping to assign RISC OS file types to files which are downloaded from the Web. If the browser does not know what to do with a particular type (there are only a small number which it knows about) and appropriate system variables have been set up then it will run the file. This will start up an application which then handles the file.

A helper application !SoundPlay has been written to play sound files in this fashion. It supplies no user interface, multitasks and supports playback of WAV, AIFF, AU, PSION, VOC, and Amiga 8SVX sound formats.

4.2.2 NC OS 1.15 - Codec play controls

Audio and video codecs (and possibly other applications) require control panels that need to be operational whilst they play and whilst the browser is still useable.

The browser provides a codec play controls toolbar as described in section 4.4.9 of the CIS. This sends messages using the plugin protocol to control the MediaPlay application that acts as an interface to the codecs themselves.

If a plugin notes that it supports these controls when starting an instance then the browser opens its play control toolbar. When the plugin closes down the toolbar is removed and replaced with the previous toolbar (as normal).

An extension to the plugin protocol allows it to be also used for helper applications ensuring that embedded and non-embedded multimedia files are treated similarly.

Special keys are included on the handset and branded keyboard to control codecs. These keys are translated by the browser into the appropriate codec commands and sent on to the appropriate plugin instance. if a plugin is highlighted then it will receive the message, otherwise the message will be sent to any suitable helper applications currently registered.

4.3 Application launching

Applications can be launched by setting up a pseudo-scheme similar to the ncfrescointernal: scheme described earlier. These URLs can then be used from simple links or from forms. In all cases the scheme name is used to determine what command to run but the full URL is always passed to the command.

If an application had set up the following alias
*Set Alias$URLOpen_FOO /<Foo$Dir> -URL %*0
then it could be used in the following ways.

4.3.1 A simple link

if the user clicks on a link of the form <A HREF="foo:myaction"> </A> then the command */<Foo$Dir> -URL foo:myaction will be run.

4.3.2 A simple form

If the user clicks submit on a form with a foo: action and METHOD=GET then the form results will be appended to the URL passed on the command line. eg

<FORM METHOD=GET ACTION="foo:myaction"> <OPTION NAME=bar CHECKED> <INPUT NAME=fred VALUE="barney" > </FORM>

When this form is submitted the command line generated will be
*/<Foo$Dir> -URL foo:myaction?bar&fred=barney

4.3.3 A complex form

If the form required will contain more data than could be fitted within the 256 character command line limit then you can use a form with a foo: action and METHOD=POST. In this case the form results will be written to a file and the filename passed on the command line. eg.

<FORM METHOD=POST ACTION="foo:myaction"> <OPTION NAME=bar CHECKED> <INPUT NAME=fred VALUE="barney" > </FORM>

When this form is submitted the command line generated will be
*/<Foo$Dir> -URL foo:myaction <temp file>
where <temp file> is a temporary file in the scrap directory. This file will contain bar&fred=barney exactly as were appended to the command line in the GET case.

4.3.4 Encoding

Note that in all cases the data, whether appended to the command line or written to a file will be encoded in the following form.

  1. All space characters are replaced by +
  2. All non alphanumeric characters are replaced by %XX where XX is the hexadecimal representation of the character code.

See the description of message OPEN URL for details on how this is handled in message form. In fact the OPEN URL message is broadcast first and only if it bounces will the URLOpen alias be used.

Note: The earlier mechanism of allowing any command to be prefixed by ncfrescointernal: has been removed for security.

4.4 Mail to (SMTP)

Not in NC 1.06

Mail to links are commonly found in HTML pages as a way of communicating with the author of the page. When clicked upon the browser should display a window for the user to enter a mail message, with the receiver's address filled in (derived from the mailto URL) and buttons to send or cancel the mail message.

Mailto links are also used in FORMs as an alternative to using the HTTP POST command for sending the form data.

4.4.1 Link to server system

If a service provider provides an email service then with the addition of some setup mailto can be handled entirely by the provider.

The browser can be configured with a mailto proxy. When a mailto link is selected the request is rerouted to this proxy (which should be a complete URL unlike the normal HTTP proxy configuration). The URL will be a cgi-bin script on the server. The mailto details are appended to the proxy URL as query information. The script can then extract this information and fill in the form appropriately.

Note that if the proxy string configured does not end with one of the three characters & = ? then the browser will append a ? before the mailto details.

Also the email address to mail to is prefixed by to= so as to make it a standard name value pair in the query string.

4.4.2 Example

The browser is configured with an mailto proxy of http://www.fred.com/cgi-bin/send-email.

If the user clicks on a link to the URL mailto:smiddleton@acorn.co.uk then the browser constructs and fetch the URL http://www.fred.com/cgi-bin/send-email?to=smiddleton@acorn.co.uk.

send-email must be a script on the server that can take the query information appended and take the user to the appropriate page in the server based email system as if they had just decided to reply to a message from smiddleton@acorn.co.uk.

4.4.3 Form example

As long as the mailto handling URL can handle form data being posted to it as well then mailto: URLs in forms can also be handled in this way.

If a form has an ACTION specified as mailto:smiddleton@acorn.co.uk and the mailto proxy is configured as in the above example, then when the SUBMIT button on the form is pressed the browser will attempt to POST the form contents to the URL http://www.fred.com/cgi-bin/send-email?to=smiddleton@acorn.co.uk. In this case the script should notice that a POST method was used and send the POSTed data (unescaped appropriately) to the email address specified. No dialogue with the user need be displayed although an HTML page saying message sent successfully or similar would be helpful - this, however, is up to the service provider.

4.4.4 User detection

The current user is sent to the server in all transactions as a cookie enabling server scripts to distinguish between different users.

4.4.5 Mailto arguments

Any extra arguments passed to the mailto: URL will be appended when this it passed to the mailto proxy. It is important that the proxy be able to handle this as this is used by the Send URL function of the browser detailed in section 29.

In the example above the url fetched if the user clicked on send url would be http://www.fred.com/cgi-bin/send-email?url=url.

4.11 Screen transitions

Not in NC 1.06

4.11.1 Introduction

Screen transitions are accomplished by the following mechanism. When a fetching page has reached the point where it is ready to start displaying then the page being displayed is faded down to the background colour of the page that is about to be displayed (which is known by this point). When the entire page has been faded down to the one colour the new page is drawn.

4.11.2 Methods

The screen fading is performed by directly accessing the screen memory to adjust the screen pixel colours. Other approaches were considered eg using the CLUTs in the video hardware, however that would fade down the toolbar as well, and redrawing the page data repeatedly, which would be very slow and incomplete as it is not possible to change the colours of images in 16bit modes.

The fade period is approximately one second.

Whilst the fading is occurring the user cannot perform any action on the page (if frames are present then no frame can be accessed) however data is still being fetched in the background so the fetching time is not affected.

4.11.3 Tiles

If there is a tiled background on the page being faded out then it is faded down along with the foreground text and images. If there is a tiled background on the new page then it is only drawn when the tile graphic is fetched. The fade colour is the background specified in the HTML. This is usually similar to the tile.

4.11.4 Frames

If the whole page is being updated then the whole page (except for the toolbar) is faded down. If only a single frame is being changed then just that frame is faded down.

If the new document being faded in is a FRAMESET then there is no background colour defined and so the default background colour (black) is used.

5.0 Programmer interface

5.1 Message Open URL (&4AF80)

The following message is supported to allow the browser to broadcast a request for another application to handle an unknown URL type or for another application application to ask the browser to render a page. This message is derived and extended from an existing mechanism in ANT Fresco. 


union
{
    char *ptr;
    int offset;
} string_value;

struct
{
    wimp_msghdr hdr;
    union
    {
        char url[236];
        struct
        {
            int tag;
            string_value url;
            int flags;
            string_value body_file;
            string_value target;
        } indirect;
    } data;
} event_data;

If the string representing the URL that needs to be opened is less than 236 bytes long and no body file or window target needs to be specified then it may be sent in the array of bytes event_data.data.url.

Otherwise the indirect form must be used. The tag value event_data.data.indirect.tag should be set to zero. Each of the 'string_value' fields above can then take one of three values.

= 0
field has no value.
< 236
field is a byte offset from the start of the message data area (&event_data.data) to the start of a string in the message body) not neceesarily word-aligned.
>= &01800000
field is a pointer to a string in shared memory (RMA or dynamic area).
Other values
reserved.

The flags are currently all reserved and should be set to zero.

The WIMP message wimp_MOPENURL, number 0x4AF80, should be broadcast with the above data, with an event code of wimp_ESENDWANTACK.

When an application receives the about WIMP message it should first determine where the URL is stored by testing the length of the string in event_data.data.url. If this is non-zero it should use the immediate string otherwise it should use the indirected strings. The application should examine the string and determine whether it can service the request. If it can service the request it should acknowledge the WIMP message before it next polls the WIMP manager. If it can not service the request it should simply ignore the message.

In the indirected case the application should also check the size of the message block to see which fields in the indirect block are valid. Older applications will broadcast the indirect request with only the tag and url fields. In this case the message will be 28 bytes long.

Note that for maximum compatibility with older receiving applications the indirected URL field should always point into the RMA as otherwise an older application will attempt to interpret the 'offset' form as a direct pointer and read from zero page which may cause a data abort.

New applications should allow for the URL being in offset or direct pointer form however for maximum flexibility.

If the requesting application receives a 'bounced' version of the WIMP message it should try to start an application to open the URL. It does so by issuing a Wimp_StartTask SWI with the command 'URLOpen_<scheme> <URL>'. If this raises an error the appliaction should notify the user that the URL could not be followed.

If no 'bounce' is received then the application should assume that the URL has been opened by some other application.

This message may be sent or handled by a number of applications. A WWW browser will send the message when the user tries to follow a link with a method that is not supported internally to the browser. The browser may also respond to the message when the method is 'HTTP'. An email or news application should examin URLs and respond to those that have a method of 'mailto' or 'news' (and maybe 'nntp'). This application may also allow the user to menu over the body of a mail or news item, try to pick out a fragment of text below the pointer that looks like a URL, and broadcast a request made from this. An FTP client should detect requests with a method of 'FTP' and service them accordingly.

Applications should, as well as accepting the WIMP message, provide a startup option that allows a URL to be specified on the command line. They should then set appropriate aliases to start the application when a URL of a known scheme is found. Thus a WWW browser should have in its !Boot file a line such as:

if "<Alias$URLOpen_HTTP>" = "" then Set Alias$URLOpen_HTTP Run <Obey$Dir> -URL %%*0

Special targets

The browser recognises a special target name __help.

If it receives an open URL request to this target then it will open a temporary browser window at the top of the window stack with the desired help page. When this window is closed (by use of the ncfrescointernal:cancel link) the caret will be returned to the window it was in when the request was sent.

A typical URL for some help pages might be file:/NCHelp:NCWriter/index.html

The favorites file is stored as a plain text file. It consists of a number of lines separated by newlines (character 0x0A). The lines of text must consist only of printable characters (32-126).

In NC100 a record consist of four lines

Line 2URL
Line 3icon name

A generic method of launching URLs from buttons is available. The event code range from 0x3100 to 0x31FF launches a URL looked up from a system variable.

7.3.5 How configuration works

7.3.5.1 Setting configuration options

Configuration of NCBrowser can occur at various times.

At startup
When the browser starts up the configuration options are used in the following order.
  1. The internal defaults are used.
  2. The file NCFresco:Config is loaded.
  3. System variables NCFresco$Config* are scanned.
At login
When a LoadUserFiles message is received.
  1. The file <NCFresco$Config> is loaded.
At other times
When a LoadConfig message is received.
  1. The file specified in the message is loaded.

At each stage the items found overwrite the previous settings but items not set keep their previous values.

The names of the system variables are the names in the Config file prefixed with NCFresco$Config and with the dots (periods) stripped out.

The internal values to NCBrowser are those used for !Fresco, the desktop version and not generally suitable for NC use. The Config file iin ROM has sensible defaults for NC use. The system variables should be used for all global overrides rather than changing the Config file.

The user Config file can be used to override certain settings on a per user basis although not all Config items can be changed at this point as the browser has already started.

Any values not specified in a Config file are left unchanged. None of the Config files need to have all or indeed any entries in them. Thus the LoadConfig message can be used to set each Config entry individually.

Example:
If you wanted to set HTTP proxying for all users on an ethernet system you would add something like the following to the !NFSBoot.!Run file. 
 Set NCFresco$ConfigProxyHttpEnable Yes
 Set NCFresco$ConfigProxyHttp 136.170.135.27:3128
 Set NCFresco$ConfigProxyHttpIgnore acorn.co.uk
Example:
If a particular user wanted to scale up their fonts by 50% then they could do the following (assuming they were using the standard server Release setup).
Create a file 'NFS::NCDHome.$.Choices.NCFresco.Config' and include in it the line
display.scale: 150

There are three other files that will be updated by the user as they browse. The file names are given by individual system variables.

NCFresco$Cookies
The users persistent cookies, updated whenever a new cookie is received or cookies are expired.
NCFresco$Plugins
The list of known plugins and whether they are disabled or enabled. Updated whenever a new plugin is found or the state is changed.

The lists of passwords. Updated whenever the user enters a new one.

7.4 Allow/Deny Files

7.4.1 Introduction

NCBrowser has very simple access controls built in.

There are two files that can be created to control which sites can be accessed, the allow file and the deny file.

They take effect whatever transport is in operation and are checked before whether to use a proxy is decided.

7.4.2 Setting the file names

The file names are read from the browser Config file. There are therefore two ways of setting them.

  1. Insert the name directly into the Config file. The two fields are files.allow and files.deny.
  2. Override the Config file entries using system variables. In this case the variable to set are NCFresco$ConfigFilesAllow and NCFresco$ConfigFilesDeny. These variables must be set before the browser starts.

These files can be either on the read-only mount or in the user's home directory. If they are in the user's home directory then they won't be accessible at startup but will be loaded when the user is logged in (along with their cookies, hotlist, etc).

7.4.3 Initialisation

On startup the browser loads the files as follows.

  1. NCBrowser starts up, reads its Config file, and reads the Config system variables.
  2. If the files.allow entry is set then it will read this file and create an allowable sites list.
  3. If the files.deny entry is set then it will read this file and create a denied sites list.
  4. When the browser is told to re read its user files (eg in Logon) then it re reads the allow and deny files. Note that although you can set up a user Config file, you cannot change the allow and deny files at this point.

7.4.4 File format

The format of both files is the same. It is plain text containing a number of lines. On each line is a hostname.

For example, if the allow file contained the following lines 

www.acorn.co.uk
www.oracle.com
www.nc.com

then the user would be able to look at only files on the acorn, oracle or nc sites.

7.4.5 Precedence

Only one of the allow/deny files is ever used at a time.

  • If the allow file is present, then everything is barred except for those sites in the file.
  • If no allow file is present but the deny file is present, then everything is allowed except for those sites in the file.

7.5 System variables

There are five classes of system variable used in NCFresco.

  1. Those hard-wired into the browser code
  2. Those only used within the !Run file
  3. Those explicitly referenced in the Config file
  4. The Config override set
  5. Non NCFresco specific variables

Some variables can be set to URLs, some to filenames, some to both. This refers to the format of the name.

  • filename means a path expressed in RISC OS style. eg. NFS::NCDBoot.$.!Boot.!Run,feb
  • url means a path expressed in URL format. This could be somewhere on the internet, eg http://www.acorn.com/ or it could actually be a file just expressed as a URL. eg the filename above would be file:/NFS::NCDBoot/$/!Boot/!Run,feb. If the variable could take either a filename or a URL then a UTL must be prefixed by -url.

Other data types are

  • filetype is expressed as a 3 character hex value. eg FB1 is the file type for Microsoft wave files.
  • command means a command that could be executed at the RISC OS command line.

10.0 Product organisation

The browser consists of a small number of support modules plus the main application itself. These are all integrated into the RISC OS build tree so that the browser can be built into the NC ROM.

In NC 1.06 the browser was build on RISC OS. Later releases must be cross-compiled on Unix and an AOF file imported into the build tree for a ROM build.

The browser is built during development via a Solaris hosted cross-compiling version of ARM cc (Acorn port) v5.06. For ROM release it is integrated into the RISC OS build tree and runs from ROM.

There are several support modules.

  • HTTP transport
  • FTP transport
  • Gopher transport
  • Resolver

These are all written in C and integrated into the build tree.

The SSLeay library that provides SSL support for the HTTP module is cross-compiled on Solaris and creates two ALF format files that are linked with the HTTP module in the main ROM build.
